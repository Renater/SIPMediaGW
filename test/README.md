# SIPMediaGW — Test Suite

This directory contains all testing and validation resources for SIPMediaGW. It is organised into three layers:

- **Environments** — Docker and Vagrant setups needed to run the tests
- **Tests** — automated API tests (pytest) and integration shell scripts
- **Simulations** — manual load/scaler simulations (not part of CI)

---

## Directory structure

```
test/
├── environments/
│   ├── vagrant/          # All-in-one VM bootstrap (Vagrantfile + provisioning script)
│   ├── baresip/          # Containerised Baresip SIP client used as a test caller
│   ├── gpu/              # GPU-enabled gateway variant for hardware-accelerated tests
│   └── proxy/            # Proxy environment (docker-compose + env config)
│
├── api/                  # Gateway API tests — pytest + shell helpers
├── proxy/                # Proxy integration tests — shell scripts
├── browser/              # Browser/WebRTC automation (navigate.py + interact.html)
└── simulations/          # Scaler load simulations — manual use only
```

## Environments

### Vagrant (all-in-one)

Spins up a single VirtualBox VM that deploys the full SIPMediaGW stack — useful for a clean end-to-end test without an existing deployment.

```bash
cd environments/vagrant
vagrant up
```

The provisioning script `all_in_one_vagrant.sh` is run automatically on first boot. To reprovision:

```bash
vagrant provision
```

To tear down:

```bash
vagrant destroy
```

> **Note:** `.vagrant/` is a runtime directory generated by Vagrant — it is not committed to version control.

---

### Baresip (SIP test client)

A containerised [Baresip](https://github.com/baresip/baresip) instance used to simulate an incoming SIP call against a running gateway.

```bash
cd environments/baresip
docker compose up -d
```

To place a test call manually:

```bash
docker exec -it baresip ./SIPCall.sh <sip-uri>
```

Configuration lives in `.baresip/`:

| File | Purpose |
|---|---|
| `accounts` | SIP account credentials |
| `config` | Baresip runtime config (ports, codecs, …) |
| `contacts` | Address book used by `SIPCall.sh` |
| `uuid` | Persistent device UUID |

> **Note:** `.baresip/current_contact` is a runtime file written by Baresip — do not commit it.

---

### GPU environment

A Docker Compose environment that runs the gateway image with GPU passthrough enabled, for validating hardware-accelerated encoding.

```bash
cd environments/gpu
docker compose up -d
```

The `config_default` file is bind-mounted into the container as the gateway configuration.

---

### Proxy environment

Deploys a SIP proxy in front of the gateway, used by the proxy integration tests.

```bash
cd environments/proxy
source env_proxy.sh        # exports required env vars
docker compose up -d
```

See `environments/proxy/README.md` for proxy-specific configuration details.

## Running the tests

### Gateway API tests (pytest)

These tests exercise the gateway's HTTP management API: starting/stopping sessions and checking status.

Run the full suite:

```bash
cd api
pytest test_http_launcher.py
```

The shell helpers in `api/helpers/` can also be called directly for quick manual checks:

```bash
# start a session
bash api/helpers/test_start.sh

# check status
bash api/helpers/test_status.sh

# send a command
bash api/helpers/test_command.sh

# stop the session
bash api/helpers/test_stop.sh
```

---

### Proxy integration tests (shell)

These scripts test the proxy layer independently. Run them against an environment started from `environments/proxy/`.

```bash
source environments/proxy/env_proxy.sh

bash proxy/test_start.sh
bash proxy/test_status.sh
bash proxy/test_admin_status.sh
bash proxy/test_command.sh
bash proxy/test_stop.sh
```

Each script exits `0` on success and prints a descriptive error on failure.

---

### Browser automation

`browser/navigate.py` drives a headless Chromium session to join a conference room, mimicking what the gateway does internally. `browser/interact.html` is a companion page for manual browser-based testing.

```bash
cd browser
python navigate.py --url https://<your-jitsi-instance>/<room>
```

## Simulations

The scripts in `simulations/` model scaler behaviour under load. They are **not** automated tests and are not run in CI — they are intended for manual capacity-planning analysis.

| File | Description |
|---|---|
| `scalerMediaSimu.py` | Simulates media gateway scaling events |
| `scalerSIPSimu.py` | Simulates SIP registration load on the scaler |
| `scaler_simu.json` | Input parameters for both simulation scripts |
| `17112020.csv` | Sample real-world traffic trace used as simulation input |
| `scaler_simulation.png` | Example output chart |

Run a simulation:

```bash
cd simulations
python scalerMediaSimu.py --config scaler_simu.json
python scalerSIPSimu.py   --config scaler_simu.json
```
